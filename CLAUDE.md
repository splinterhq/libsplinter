## What This Project Is

Splinter is a high-performance, lock-free, shared-memory key-value store and message bus designed for low-latency IPC without kernel overhead. It targets LLM inference deployments where tight coupling between processes is needed. Key design constraints: passive substrate (no daemon required), zero-copy via mmap, atomic sequence locks per-slot, static geometry (pre-allocated arena), and epoll-based signal pulse system.

## Build Commands

```bash
./configure [options]   # Generate build/ directory (wraps CMake)
make                    # Build from top level (shim routes to build/Makefile)
make prod               # Production build
make tests              # Run all tests via ctest
make clean              # Clean build dir
make distclean          # Full git clean
sudo -E make install    # Install
```

Configure options:
```bash
./configure --with-embeddings   # KV store with embeddings
./configure --with-llama        # LLM inference support
./configure --with-lua          # Lua script integration
./configure --with-wasm         # WasmEdge runtime
./configure --with-rust         # Rust bindings
./configure --with-numa         # NUMA affinity
./configure --dev               # All features enabled
```

Compiled with `-O3 -Wall -Wextra -D_GNU_SOURCE -fPIC` (C11 standard).

## Running Tests

```bash
cd build && ctest --output-on-failure   # All tests
cd build && ./splinter_test             # Unit tests (standard memory)
cd build && ./splinterp_test            # Unit tests (persistent mode)
cd build && ./splinter_stress --duration-ms 7500   # Multi-threaded stress test
cd build && ./splinterctl_tests.sh      # CLI regression tests

# Run a named test with verbose output
cd build && ctest -R "standard_unit_test" -V

# Memory leak checks (if valgrind installed — also runs via ctest automatically)
valgrind -s --leak-check=full ./splinter_test
```

The persistent variant (`splinterp_test`) is compiled from the same `splinter_test.c` with `-DSPLINTER_PERSISTENT`. Valgrind tests are auto-registered in CMakeLists.txt when valgrind is found.

## Architecture

**Core:**
- `splinter.h` / `splinter.c` — 33KB public API + ~890-line lock-free KV store core. All store logic: mmap lifecycle, slot management, atomic ops, signal lanes.
- The persistent variant is the same `splinter.c` compiled with `-DSPLINTER_PERSISTENT` for file-backed storage.

**CLI (`splinterctl`):**
- `splinter_cli_main.c` + `splinter_cli.h` — REPL and one-shot command framework.
- `splinter_cli_cmd_*.c` — 25+ individual command modules (get, set, watch, label, math, lua, wasm, etc.).
- `splinter_cli_util.c`, `splinter_cli_tok.c` — CLI utilities and tokenizer.

**Inference:**
- `splinference.cpp`, `splainference.cpp` — Quantized Nomic embedding engine (llama.cpp integration).
- `sidecar.c` — System observer/monitoring daemon with ring buffer output.

**Bindings:**
- `bindings/rust/` — Cargo workspace with `memory` and `persistent` crates (FFI bindings).
- `bindings/ts/` — TypeScript/Bun bindings.

**3rdparty:**
- `linenoise.c/h`, `argparse.c/h`, `libgrawk.c/h` — Never modify these.

**Key data structures** (defined in `splinter.h`):
- `splinter_header` — Store metadata (magic, version, slot count, capacity).
- `splinter_slot` — Per-key metadata (hash, epoch, access/change times, bloom flags, type).
- Value arena — Fixed-size contiguous storage; no heap allocation at runtime.
- Signal groups — 64 independent epoll-based notification lanes.

**Type system:** VOID, BIGINT, BIGUINT, JSON, BINARY, IMGDATA, AUDIO, VIDEO, VARTEXT.

## Important Constraints

- **Valgrind-clean is a hard requirement.** All commits must be free of memory leaks and errors. Run `make tests` before submitting — ctest includes Valgrind passes when available.
- **Core LOC budget:** Keep `splinter.c` under 1,000 lines of real logic. The project deliberately maintains mechanical simplicity.
- **Linux-only.** The codebase uses POSIX mmap, epoll, `stdatomic`, and GNU extensions (`_GNU_SOURCE`). No portability shims.
- **No dynamic heap at runtime.** The arena is statically allocated at init; do not introduce `malloc`/`free` into hot paths.
- **`splinterrc_example`** is installed to `~/.splinterrc` and documents bloom filter configuration. The bloom system is user-configurable, not hard-coded.

## Configuration & Generated Files

- `config.h` and `build.h` are generated by CMake — do not edit them directly.
- `CMakeLists.txt` owns the full feature matrix, test registration, and install targets.
- `configure` is a 66-line shell wrapper — keep it simple.
- `Makefile` at the top level is a shim; real build logic is in `build/Makefile` (CMake-generated).
